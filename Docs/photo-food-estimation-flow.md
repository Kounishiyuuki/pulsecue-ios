# 写真による食事推定フローの設計（Photo Food Estimation Flow）

> **本ドキュメントの目的**: 食事写真から AI でカロリー / タンパク質を推定する機能を
> 実装する **前に**、安全な挙動・データフロー・実装境界を確定するための設計資料。
> **実装変更は含まない**（ドキュメントのみ）。AI 連携・写真アップロード・ネットワーク
> 通信・UI 実装・スキーマ変更は本 PR では一切行わない。
>
> 対象読者: 後続の実装 PR を切る開発者。
> 作成日: 2026-05-21。

関連する既存仕様:
[`ai-privacy-and-safety.md`](ai-privacy-and-safety.md)（AI 安全境界の鉄則）、
[`credential-strategy.md`](credential-strategy.md)（クライアントに長期キーを置かない方針）。

## 1. 目的（Purpose）

### なぜ写真食事推定が必要か

PulseCue の既存の食事入力（手動 / バーコード / 栄養表示 OCR）は、いずれも
**数値が手元にある**ことを前提とする。一方で現実の食事の多くは:

- 外食・自炊で、栄養成分表示もバーコードも存在しない。
- 複数の料理が 1 皿に乗っており、品目ごとの数値が分からない。
- ユーザーが栄養成分を調べる手間をかけたくない。

写真食事推定は、この「数値が手元にない食事」をできるだけ低摩擦で記録するための
**最後のフォールバック導線**として位置づける。

### なぜバーコード / OCR より信頼性が低いか

写真からの推定は、本質的に次の不確実性を含むため、バーコード照会や栄養表示 OCR
よりも誤差が大きい:

- **分量（ポーション）**: 画像だけでは皿のサイズ・盛り付け量を正確に推定できない。
- **材料・隠れた成分**: 油・ソース・ドレッシング・砂糖など、見た目に現れない高カロリー
  成分を推定できない。
- **調理法**: 「揚げ」「焼き」「蒸し」でカロリーが大きく変わるが画像から判別しにくい。
- **重なり・遮蔽**: 料理が重なっていると品目や量を取りこぼす。

したがって写真推定は **確定値ではなく「候補（candidate）」を生成する機能**であり、
UI コピーでも「推定値は誤っている可能性がある」ことを明示する。バーコード /
OCR が使える場面ではそちらを優先する導線設計とする。

## 2. 既存の食事入力ハイアラーキー

PulseCue には現在 5 つの食事入力経路がある。写真推定はその最下位に**追加**される
（既存経路を置き換えない）。信頼性が高い順:

| 経路 | ソース | 信頼性 | 確定方法 |
|---|---|---|---|
| 手動入力（`MealEntrySheet`） | `.manual` | ユーザーが数値を直接入力 | 保存即 `.confirmed` |
| 最近の食事（PR #40） | `.manual` | 過去の確定値の再利用 | タップで即 `.confirmed` |
| よく使う食事テンプレート（PR #42） | `.manual` | ユーザーが固定した確定値 | タップで即 `.confirmed` |
| バーコード商品照会（PR #45） | `.barcode` | 商品 DB 由来（Open Food Facts） | レビュー画面で確認後 `.confirmed` |
| 栄養表示ラベル OCR（PR #46） | `.ocr` | 端末内 Vision で表示値を読取 | レビュー画面で確認後 `.confirmed` |
| **写真食事推定（本ドキュメント）** | **`.ai`** | **最も不確実**（推定） | **レビュー画面で確認後 `.confirmed`** |

写真推定はソース `.ai` を用いる（`MealSource.ai` は `AICoachStub.swift` 用に既に定義済み）。
バーコード / OCR と同じく「**候補を出す → ユーザーがレビューして確定する**」二段構えとし、
スキップや自動確定は設けない。

## 3. 中核の安全ルール（Core Safety Rule）

[`ai-privacy-and-safety.md`](ai-privacy-and-safety.md) の鉄則を写真推定に適用したもの。
**後続のすべての実装 PR はこのルールに従わなければならない。**

1. **AI / 写真推定の結果は「候補」にすぎない。** 推定そのものは事実でも記録でもない。
2. **候補は DayLog を自動更新してはならない。** `DayLog.intakeCalories` や
   タンパク質合計に、確認前の推定値が混ざることは一切許されない。
3. **候補は、ユーザーがレビューして確定した後にのみ `.confirmed` の `MealEntry`
   になる。** 既存の `NutritionLedger` の「`.confirmed` のみが合計に加算される」
   ルール、および `ProteinTotals` の「`.confirmed` のみ集計」ルールが、写真推定でも
   そのまま安全装置として効く。
4. **暗黙のネットワーク送信ゼロ。** 写真が AI プロバイダへ送られるのは、ユーザーが
   その画面でその操作（「推定する」ボタン等）を明示的に行ったときだけ。
5. **既存の `UserConfirmed<MealEstimate>` 境界を経由する。** AI が生成した値を
   モデルへ書き込むときは `AICoachStub.swift` の `UserConfirmed` でラップし、
   `applyConfirmedMealEstimate(_:to:)` 相当の単一経路を通す。ラップなしの
   DayLog 直書きは禁止。

## 4. 提案するフロー（Proposed Flow）

```
写真撮影 / 写真選択
  → （ユーザーが「推定する」を明示タップ）
  → AI 推定リクエスト（オプトイン時のみ・1 回限り）
  → 候補結果（NutritionEstimateCandidate）
  → レビュー画面（栄養表示 OCR レビュー画面と同型）
  → ユーザーが 食事名 / カロリー / タンパク質 / 時間帯 / 分量メモ を編集
  → 「食事として保存」を確定タップ
  → MealEntry(status: .confirmed, source: .ai) を 1 件作成（今日のローカル日付）
  → NutritionLedger.syncDayLogIntake(for: today)
```

- 写真の取得は端末内（撮影 / フォトライブラリ選択）で完結する。**取得しただけでは
  推定もネットワーク通信も発生しない。**
- 推定リクエストはユーザーの明示操作 1 回につき 1 回。バックグラウンド推定なし。
- レビュー画面は PR #45 / #46 の `BarcodeProductReviewView` /
  `NutritionLabelReviewView` と同じ構造（`Form` ベース、確定ボタンは
  カロリーが有効値のときのみ活性）を再利用または踏襲する。
- 確定（confirm）でのみ `MealEntry` を作成し、その直後に `NutritionLedger` を同期する。
- キャンセル / 戻るでは `MealEntry` を作らず、DayLog にも触れない。

## 5. ペンディングモデルの扱い（Pending Model）

`MealStatus` には `.pending` と `.confirmed` がある。写真推定の候補をどう保持するかに
2 つの選択肢がある。

### 選択肢 A: ペンディング `MealEntry` を作る（既存 AI コーチ導線と同型）

`NutritionView` の現在の AI 推定カードは `.pending` の `MealEntry` を作り、画面に
カードとして表示し、ユーザーが「確定」して初めて `.confirmed` になる。

- 利点: 推定結果がアプリ再起動後も残る。
- 欠点: 確定し忘れたペンディング行が溜まる。`.pending` が誤って合計に混ざる経路を
  作らないよう常に注意が必要。レビュー UI とリスト UI の二重管理になる。

### 選択肢 B（推奨）: 候補をビュー状態に保持し、確定時のみ 1 件作成

PR #45（バーコード）/ PR #46（OCR）が採用した新しいパターン。候補は
`NutritionEstimateCandidate` のような **値型**としてレビュー画面の状態に保持し、
ユーザーが確定したときに初めて `.confirmed` の `MealEntry` を 1 件作る。

- 利点: ペンディング行が一切生まれない。「確認前の値が合計に混ざる」リスクが
  **構造的にゼロ**（未確定の候補はそもそも SwiftData に存在しない）。レビュー画面が
  単一の責務を持ち、テストも純粋関数（`makeConfirmedEntry` 相当）で書ける。
- 欠点: 推定結果はレビュー画面を離れると失われる（再推定が必要）。写真推定は
  クラウド課金を伴う可能性があるため「離脱したら結果が消える」のは UX 上もむしろ
  自然で、許容できる。

### 推奨

**MVP は選択肢 B を採用する。** 理由:

- 安全ルール（3 章）を最も単純に保証できる。`.pending` の写真推定行が存在しないので、
  `NutritionLedger` / `ProteinTotals` のルールを変更する必要が一切ない。
- バーコード / OCR と実装・テスト・UX が揃い、レビュー画面を 3 経路で共通化しやすい。
- 既存の AI コーチカード（`.pending` 利用）はそのまま残してよい。写真推定は別導線として
  選択肢 B で独立して始める。

なお選択肢 A を選ぶ場合でも、**`.pending` は絶対に合計へ加算しない**という既存ルールは
不変であり、`NutritionLedger` / `ProteinTotals` の変更は依然として不要。

## 6. レビュー画面に必要なフィールド

`NutritionLabelReviewView`（PR #46）を基準に、写真推定向けに以下を備える:

| フィールド | 必須 | 備考 |
|---|---|---|
| 食事名 | 任意 | 空なら時間帯ラベルにフォールバック（既存 `makeConfirmedEntry` の挙動） |
| カロリー（kcal） | **必須** | 有効な非負整数になるまで確定ボタンを無効化 |
| タンパク質（g） | 任意 | 空のままでも確定可。値があれば `ProteinTotals` 経由で集計される |
| 時間帯（スロット） | 必須 | `MealSlot` ピッカー |
| 分量 / 量のメモ | 任意 | 「ご飯 大盛り」等。`MealEntry.note` を流用（スキーマ追加不要） |
| 信頼度 / 警告テキスト | 表示のみ | 「この推定値は誤っている可能性があります。保存前に確認してください。」を常時表示。
  推定が低信頼または複数候補のときは追加の注意文を出す |

- 炭水化物 / 脂質は任意フィールドとして OCR レビュー画面と同様に提供してよい。
- **新しい永続フィールドは追加しない。** 分量・信頼度メモはすべて既存の
  `MealEntry.note`（`String?`）またはビュー状態で表現し、SwiftData スキーマは変更しない。

## 7. エラー / フォールバック状態

各状態で **MealEntry を作らず、DayLog を更新せず**、ユーザーに次の行動を明示する。

| 状態 | UI の振る舞い |
|---|---|
| 画像が不鮮明 | 「写真が不鮮明です」＋ 別の写真を試す / 手動入力へ |
| 食べ物を認識できない | 「料理を認識できませんでした」＋ 手動入力フォールバック |
| 複数の料理を検出 | 候補を提示し、ユーザーが対象を選ぶ。確定するまで何も保存しない |
| カロリー欠落 | レビュー画面で手動入力。カロリーが有効値になるまで確定ボタン無効 |
| タンパク質欠落 | 空のまま確定可、または手動入力 |
| ネットワーク失敗 | 「推定できませんでした。通信状況を確認してください」＋ 再試行 / 手動入力 |
| モデル利用不可（AI 無効 / 未対応） | 「写真推定は現在利用できません」＋ 手動・バーコード・OCR 導線を案内 |
| ユーザーがキャンセル | 何も作らず画面を閉じる。副作用ゼロ |

すべてのフォールバックは最終的に **手動入力**へ着地できること。写真推定が失敗しても
ユーザーが食事を記録する手段を失わない。

## 8. プライバシーとデータの取り扱い

[`ai-privacy-and-safety.md`](ai-privacy-and-safety.md) に従う。写真は特に機微なので追加で:

- 写真が AI プロバイダへ送られるのは、ユーザーがその画面で推定を**明示的に開始**した
  ときだけ。自動アップロードは禁止。
- 写真撮影 / 選択の時点ではネットワーク通信を発生させない（取得とアップロードを分離）。
- **生写真を永続化しない**（明確な必要が出るまで）。推定後はメモリ上の画像を破棄し、
  `MealEntry` には数値・名前・メモのみを保存する。サムネイル保存が必要になった場合は
  別途スキーマ設計の検討が必要（本 PR 範囲外）。
- 画像・推定リクエスト / レスポンスのペイロードを**ログに残さない**（エラー時も含む）。
  テレメトリにユーザー入力・画像を含めない。
- **API キーをアプリバイナリに埋め込まない。** クライアントに長期キーを置く構成は禁止
  （[`credential-strategy.md`](credential-strategy.md) の脅威モデル参照）。
- AI プロバイダの認証が必要な場合は、クライアントシークレットではなく
  **セキュアなバックエンド / 短期トークン方式**を用いる。具体仕様は
  `credential-strategy.md` の案 B（短期デバイススコープトークン + App Attest）と
  [`import-token-endpoint-spec.md`](import-token-endpoint-spec.md) を参照し、
  写真推定向けの相当エンドポイントは後続 PR で別途設計する。
- 設定のオプトインは細粒度に。「食事カロリー推定を有効にする」トグル（既存
  `ai-privacy-and-safety.md` の TODO）に写真推定もぶら下げ、デフォルト OFF。

## 9. コストとオフラインの考慮

- 写真からの食事推定は、実用精度を出すにはクラウド / 課金推論が必要になる可能性が高い
  （端末内モデルだけで十分な精度が出るかは実装 PR で要検証）。
- **AI はあくまでオプション。** 手動入力・最近の食事・よく使う食事・バーコード照会・
  栄養表示 OCR は、AI が無効でもネットワーク不通でも従来どおり完全に動作し続ける
  こと。写真推定だけが AI / ネットワークに依存する。
- AI が無効・未対応・課金未設定の場合は、写真推定の導線を「利用不可」として明示し、
  他の入力経路へ誘導する（7 章「モデル利用不可」）。
- 推論コストが課金に直結するため、推定はユーザー操作 1 回につき 1 リクエストに限定し、
  リトライもユーザー明示操作に限る（暗黙の再送なし）。

## 11. 明示的な非ゴール（Non-Goals）

写真食事推定は次を**やらない**:

- 医療・栄養学的な診断やアドバイス。
- カロリーの自動コミット（確認なしで DayLog に加算すること）。
- 推定精度の保証。「正確な値」をうたわない。UI で誤差の可能性を明示する。
- 写真の隠れたアップロード。ユーザーの明示操作なしに画像を送らない。
- クライアント（アプリバイナリ）への API キー埋め込み。
- 既存の食事入力経路（手動 / 最近 / テンプレ / バーコード / OCR）の置き換え。
- `NutritionLedger` / `ProteinTotals` の集計ルール変更。
- SwiftData スキーマの変更。

## 12. 将来実装の受け入れ基準（Acceptance Criteria）

PR F（実統合）以降が満たすべき条件。1 つでも破ったら実装は不合格とする:

- [ ] 推定値は、確認前に DayLog / 摂取カロリー / タンパク質合計を変えない。
- [ ] キャンセル / 戻るは何も作らない（`MealEntry` ゼロ、DayLog 不変）。
- [ ] 確定は `.confirmed` の `MealEntry` を **ちょうど 1 件**作る（`source: .ai`、
      今日のローカル日付）。
- [ ] `NutritionLedger.syncDayLogIntake` は確定後にのみ呼ばれる。
- [ ] AI 推定は `UserConfirmed<MealEstimate>` 境界を経由する。
- [ ] ユニットテストはモック / フィクスチャを使い、ライブ AI を呼ばない（パーサや
      候補→`MealEntry` 変換は純粋関数として強くテストする）。
- [ ] 写真が送られるのはユーザーの明示操作時のみ。暗黙アップロードなし。
- [ ] API キーがアプリバイナリ・リポジトリ・ログに現れない。
- [ ] AI 無効 / オフラインでも手動・バーコード・OCR 経路が従来どおり動く。
- [ ] ユーザー向けコピーが「推定値は誤っている可能性がある」ことを明示する。
