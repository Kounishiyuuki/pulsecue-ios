# 写真 AI プロバイダ / クレデンシャル戦略（Photo AI Provider Strategy）

> **本ドキュメントの目的**: 食事写真からの**実 AI 推定**を実装する **前に**、
> プロバイダの選択肢・クレデンシャルの扱い・プライバシー規則・コスト・実装順序を
> 確定するための設計資料。
> **実装変更は含まない**（ドキュメントのみ）。実 AI 連携・ネットワーク通信・写真
> アップロード・API キー処理・バックエンド変更・UI 実装は本 PR では一切行わない。
>
> 対象読者: 実 AI 写真推定を担当する後続 PR の開発者。
> 作成日: 2026-05-22。

関連する既存仕様:

## 1. 目的（Purpose）

[`photo-food-estimation-flow.md`](photo-food-estimation-flow.md) は写真推定の
**アプリ内フロー**（写真 → 候補 → レビュー → 確定 → 保存）と安全境界を定義した。
PR #48–#50 でそのフローはモックのみで実装済み（capture プロトタイプ、モック候補、
`PhotoFoodEstimating` 抽象化）。

しかし**実 AI** を足すには、フロー設計とは別に次を決めなければならない:

- 推論をどこで行うか（端末内 / 自社バックエンド経由 / プロバイダ直叩き）。
- プロバイダの認証情報（API キー等）をどう扱うか。
- 写真という機微データをどこへ、いつ送るか。
- 課金・レート制限・障害時の挙動。

これらは**クレデンシャルとプライバシーの問題**であり、コードを書く前に確定すべき
意思決定である。本ドキュメントはその意思決定資料。実 AI は、本戦略が承認されるまで
実装しない。

## 3. 中核ルール（Core Rules）

[`ai-privacy-and-safety.md`](ai-privacy-and-safety.md) の鉄則を写真 AI に適用したもの。
**後続のすべての実装 PR はこれを守らなければならない。**

1. **iOS アプリに AI プロバイダの API キーを同梱しない。** 公開バイナリは展開・逆
   アセンブル可能であり、埋め込んだキーは抽出される（[`credential-strategy.md`](credential-strategy.md) 2 章の脅威モデル）。
2. **写真の隠れたアップロードをしない。** 画像が外部へ送られるのは、ユーザーがその
   画面でその操作（「推定する」等）を明示的に行ったときだけ。
3. **DayLog を自動更新しない。** 推定結果が確認前に摂取カロリー / タンパク質合計へ
   混ざることは一切許されない。
4. **推定は「候補」にとどまる。** 実 AI の結果も `PhotoFoodEstimate` 候補にすぎず、
   事実でも記録でもない。
5. **確定済み `MealEntry` の前に必ずユーザーレビューを挟む。** `PhotoEstimateReviewView`
   での明示確定（「食事として保存」）のみが保存経路。
6. **キャンセルは何も作らない。** レビュー画面を離脱したら候補は破棄、副作用ゼロ。

## 4. プロバイダ方式の候補（Candidate Provider Approaches）

実 AI 推論を「どこで・どう」行うかの 4 案を比較する。

| | **A. iOS → プロバイダ直叩き** | **B. iOS → PulseCue バックエンド → プロバイダ** | **C. 端末内モデル** | **D. 手動のみ（写真 AI を入れない）** |
|---|---|---|---|---|
| **概要** | アプリが AI プロバイダ API を直接呼ぶ。キーはアプリ内 | アプリは自社バックエンド/Worker を呼び、そこがプロバイダを呼ぶ。プロバイダキーはサーバ側のみ | Core ML 等で端末内推論。外部通信なし | 写真 AI を実装せず、手動/バーコード/OCR のみ |
| **セキュリティ** | 不可。長期キーがバイナリから抽出される | 良。プロバイダキーは iOS に降りてこない。iOS は短期トークンのみ保持 | 最良。秘密情報も画像送信も無い | 最良（機能なし=リスクなし） |
| **コスト** | プロバイダ課金が直接発生。乱用を止められない | プロバイダ課金 + バックエンド運用。サーバ側でレート制限・クォータ管理が可能 | 推論コストはゼロ。モデル同梱でアプリ容量増 | ゼロ |
| **プライバシー** | 写真がプロバイダへ直行。経路の制御が弱い | 写真の送信先・保持を自社境界で制御できる。ログ最小化を強制しやすい | 写真が端末外に出ない。最良 | 写真送信なし |
| **実装労力** | 小（ただし不可なので採らない） | 中〜大（iOS の `TokenProvider` + バックエンド proxy） | 大（モデル選定・精度検証・容量最適化） | 0 |
| **App Store 適合性** | キー埋め込みは規約・セキュリティ上不適 | 適 | 適 | 適 |
| **本番採用可否** | **不可** | **可（実クラウド AI を入れる場合の本命）** | 可（端末内で十分な精度が出るなら理想） | 可（写真 AI を見送る場合） |

補足:

- **A** は検討対象から外す。`#if DEBUG` ガードを付けても、写真 AI は公開機能であり
  デバッグ専用では意味がない。[`credential-strategy.md`](credential-strategy.md) が
  ジム取り込みについて出した結論（長期キーを公開ビルドに入れない）と同じ。
- **B** は [`credential-strategy.md`](credential-strategy.md) の案 B（短期デバイス
  スコープトークン + App Attest）と同型。プロバイダキーは Worker secret として
  サーバ側にのみ存在し、iOS には短期トークンだけが降りる。
- **C** は精度が最大の不確実要素。食事写真からの分量・調理法推定は端末内モデルで
  実用精度に届くか未検証（[`photo-food-estimation-flow.md`](photo-food-estimation-flow.md) 1 章の不確実性）。
  実現すれば最もプライバシーが良いので、後続 PR で評価する価値はある。
- **D** は常に有効な土台。写真 AI が無くても手動 / バーコード / 栄養表示 OCR で
  食事は記録できる。

## 5. 推奨アプローチ（Recommended Approach）

- **iOS にプロバイダ API キーを直接置かない（案 A を不採用）。** 例外なし。
- **実クラウド AI を入れるなら案 B（バックエンド / トークン仲介）。** プロバイダ
  キーは自社境界にのみ置き、iOS は短期トークンで認証する。具体仕様は
  [`credential-strategy.md`](credential-strategy.md) 案 B と
  [`import-token-endpoint-spec.md`](import-token-endpoint-spec.md) を写真推定向けに
  別 PR で具体化する。
- **案 C（端末内モデル）は将来の評価対象として残す。** プライバシー上は理想なので、
  精度が出るなら案 B より望ましい。実装着手前に PoC で精度を確認する。
- **手動 / バーコード / OCR は AI 無しで常に使えるようにする。** 写真 AI はあくまで
  オプション。AI が無効・課金未設定・オフラインでも他経路は従来どおり動く。
- **モックプロバイダ（`MockPhotoFoodEstimator`）はテスト / オフライン開発用に残す。**
  実プロバイダ追加後も `PhotoFoodEstimating` のモック実装は削除しない。
- **クレデンシャル戦略が確定するまで実 AI を実装しない。** 本ドキュメントと後続の
  バックエンド設計 PR（§11）が承認されるまで、写真推定はモックのみで進める。

## 6. リクエスト / レスポンス境界（Request/Response Boundary）

将来の実プロバイダが満たすべき**概念上の**入出力契約。具体的なメソッドシグネチャは
実装 PR（§11 PR 52）で `PhotoFoodEstimating` を拡張して確定する。

**入力:**

- 画像データ、または推定用に前処理した画像ペイロード（リサイズ / 圧縮済み）。
- 任意のメタデータ: 食事の時間帯（スロット）、ユーザーメモ、ロケール（`ja` 等）。
- メタデータは推定の手がかり用。**個人を特定しうる文脈は送らない**（§7）。

**出力（`PhotoFoodEstimate` 候補 — 既存の型を拡張しうる）:**

- 食事名
- カロリー（kcal）
- タンパク質（g）
- 任意: 炭水化物 / 脂質（g）
- 任意: 信頼度（confidence）または警告テキスト
- 任意: モデル / プロバイダのメタデータ（**秘密情報を含まない範囲で**。デバッグ用）

**境界の鉄則:**

- プロバイダの**シークレットがクライアントに露出しない**こと。iOS が扱うのは短期
  トークンまで（案 B）。
- 出力はあくまで候補。`PhotoFoodEstimating` の戻り値は `PhotoFoodEstimate` のままで、
  実プロバイダでもモックでも同じ型をレビュー画面へ渡す。
- 現行 `PhotoFoodEstimating.estimate()` は画像入力を取らない。実装 PR で画像入力を
  追加する際も、レビュー / 保存フロー（`PhotoEstimateReviewView` 以降）は不変に保つ。

## 7. プライバシー規則（Privacy Rules）

[`ai-privacy-and-safety.md`](ai-privacy-and-safety.md) に従う。写真は特に機微なので追加で:

- **写真のアップロードはユーザーの明示操作後のみ。** 撮影 / 選択しただけでは送信
  しない（取得とアップロードを分離）。
- **バックグラウンドアップロードをしない。**
- **生写真をデフォルトで永続化しない。** 推定後はメモリ上の画像を破棄する。サムネイル
  等の保存が必要になったら別途スキーマ設計の検討が要る（本ドキュメント範囲外）。
- **画像データをログに残さない。** エラー時・テレメトリでも画像を出力しない。
- **食事ペイロードを個人的に機微な文脈と一緒にログしない。** 位置・健康履歴・体重
  などを推定リクエストに同梱しない。
- **ユーザー向けコピーが推定の不確実性を明示する。** 「推定値は誤っている可能性が
  ある／保存前に確認」を画面に出す（既存のモックレビュー画面の文言を踏襲）。
- オプトインは細粒度。「食事カロリー推定を有効にする」トグル（既定 OFF）に写真 AI を
  ぶら下げる。

## 8. コストとレート制限（Cost and Rate Limiting）

- 実クラウド AI は**従量課金**になりうる。乱用・暴走は直接コストになる。
- 公開リリース前に**レート制限**が必要。案 B ならバックエンド側でトークン単位の
  クォータを設ける（[`credential-strategy.md`](credential-strategy.md) PR-δ 相当）。
- **クォータ超過 / ネットワーク障害時のフォールバック**を用意する（§9）。最終的に
  手動入力へ着地できること。
- **リトライはユーザーの明示操作に限る。** 失敗時の自動再送をしない。
- **同一操作で推論を繰り返さない。** ユーザー操作 1 回 = 推定 1 回。

---

## 9. エラー / フォールバック状態（Error and Fallback States）

各状態で **MealEntry を作らず、DayLog を更新せず**、次の行動を明示する。

| 状態 | 振る舞い |
|---|---|
| ネットワーク障害 | 「推定できませんでした。通信状況を確認してください」+ 再試行 / 手動入力 |
| プロバイダタイムアウト | 同上。リトライはユーザー操作のみ |
| 不正 / 非対応の画像 | 「この画像は処理できません」+ 別画像 / 手動入力 |
| 料理を認識できない | 「料理を認識できませんでした」+ 手動入力フォールバック |
| カロリー欠落 | レビュー画面で手動入力。カロリーが有効値になるまで確定不可 |
| タンパク質欠落 | 空のまま確定可、または手動入力 |
| プロバイダ利用不可（AI 無効 / クォータ切れ） | 「写真推定は現在利用できません」+ 手動 / バーコード / OCR を案内 |
| ユーザーがキャンセル | 何も作らず画面を閉じる。副作用ゼロ |

すべてのフォールバックは最終的に**手動入力**へ着地できること。

## 10. テスト戦略（Test Strategy）

- **モックプロバイダのテストを基本とする。** `MockPhotoFoodEstimator` /
  `PhotoFoodEstimating` の既存テストを土台に拡張する。
- **ユニットテストはライブ AI プロバイダを既定で叩かない。** フィクスチャ / モックを
  使う（バーコード `OpenFoodFactsProductLookupService` のスタブ `URLProtocol`、OCR の
  フィクスチャ文字列と同じ方針）。
- **保存フローのテストで「確認前に DayLog が変わらない」ことを検証する。** 既存の
  `PhotoEstimateReviewSaveTests`（確定前は候補が不活性、確定で `.confirmed .ai` の
  `MealEntry` 生成 + `NutritionLedger` 同期）を維持・拡張する。
- **セキュリティ grep チェック**: 各 PR の差分に API キー / プロバイダ URL /
  `Authorization: Bearer` / `wrangler` / `.workers.dev` 等が現れないことを確認する。
- ライブプロバイダに対する結合テストを行う場合は、明示的に opt-in されたもののみ・
  既定の CI では走らせない。

## 12. 明示的な非ゴール（Non-Goals）

本 PR および写真 AI 戦略は次を**やらない / 認めない**:

- iOS クライアントへのプロバイダ API キー直接埋め込み。
- 写真の隠れたアップロード。
- カロリーの自動コミット（確認なしの DayLog 加算）。
- 医療・栄養学的な診断やアドバイス。
- 推定精度の保証。
- 本 PR での実 AI 実装 / OpenAI 連携 / ネットワーク通信 / 写真アップロード。
- 本 PR でのコード・スキーマ・`server/` 変更。

## 13. 実 AI 実装前の受け入れ基準（Acceptance Criteria）

PR 55（実プロバイダ統合）に着手してよいのは、次がすべて満たされたとき:

- [ ] プロバイダ方式（§4 のいずれか。本命は案 B）が承認されている。
- [ ] クレデンシャル経路が承認されている（iOS は短期トークンまで）。
- [ ] クライアント側にシークレットが存在しない設計になっている。
- [ ] 写真アップロードに関するユーザー向けコピーが承認されている。
- [ ] レビュー前確定なし（review-before-save）の境界が保たれている。
- [ ] テストが既定でモックを使う（ライブ AI を叩かない）。
- [ ] 手動 / バーコード / OCR のフォールバックが引き続き利用できる。

1 つでも未達なら実 AI 実装は着手不可。それまで写真推定はモックのみで進める。
